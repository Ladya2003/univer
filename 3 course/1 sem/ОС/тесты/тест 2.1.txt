Примечание. Ответ легко найти в интернете, но рекомендую выполнить все битовые преобразования самостоятельно. Алгоритм можно взять хоть с википедии, страница UTF-8.

Для преобразования символа из Unicode (U+042E для Ю) в UTF-8, следуем алгоритму для кодов из диапазона U+0800 – U+FFFF:

Символ кодируется 16-битным числом: 042E

Это число занимает 3 байта в UTF-8:

1110xxxx 10xxxxxx 10xxxxxx
Разбиваем 042E на биты:

0100 0010 1110
Распределение битов по шаблону UTF-8:

11100100 10001011 10101110 (это просто вставка наших битов в шаблон)
Преобразование в шестнадцатеричную систему счисления:

E4 8B AE
Итак, правильный ответ: E4 8B AE. Однако этого варианта нет среди предложенных ответов.

В Windows не существует жёсткого ограничения на количество дочерних процессов для одного родительского процесса. Ограничение может наступить из-за доступных системных ресурсов (память, дескрипторы и т.д.), но не из-за искусственного лимита на количество дочерних процессов. Поэтому правильный ответ:

Может.

Что входит в контекст процесса? Выберите ВСЕ подходящие варианты.
Контекст стека
Контекст ядра
Пользовательский контекст
Регистровый контекст
Системный контекст
  
Контекст процесса — это набор значений, которые определяют процесс в любой конкретный момент времени и которые сохраняются, когда процесс переводится в состояние ожидания и восстанавливаются, когда процесс активизируется. В контекст процесса включается:

Контекст стека: Содержимое стека процесса, которое включает локальные переменные и информацию для возврата.

Контекст ядра: Информация, которая ядро ОС хранит для процесса, такая как дескрипторы открытых файлов, состояние привилегий и другие системные атрибуты.

Пользовательский контекст: Этот контекст относится к области памяти процесса, которая содержит код, данные и стек процесса.

Регистровый контекст: Значения процессорных регистров.

Системный контекст: В зависимости от контекста и ОС, это может включать состояние памяти, планировщика и другие системные ресурсы.

Таким образом, все перечисленные варианты входят в контекст процесса.

Правильный порядок для отображения файла в памяти в Windows API:

CreateFile - открывает существующий или создает новый файл.
CreateFileMapping - создает или открывает именованный или бесименный объект отображения файла.
MapViewOfFile - отображает просмотр объекта отображения файла в адресное пространство вызывающего процесса.
Таким образом, правильный порядок: 1-2-4.
